# Development Process

開発プロセス（修正作業の実施手順等）に関する共通ルールです。

## 概要

このルールは、修正作業の実施手順、原因調査と分析、修正方針の提示方法を定義します。効率的で安全な開発プロセスを実現するための基本原則です。

## 基本原則

### 1. 修正作業の実施手順

**🚨 重要な実装ルール**:

**絶対に守るべきルール**:

1. **原因調査と分析を先に行う**
   - 問題が発生した場合、まず原因を特定する
   - 現状を詳しく調査・分析する
   - 問題の影響範囲を明確にする

2. **修正方針を提示して承認を得る**
   - 原因分析結果を報告する
   - 複数の修正方針（Option 1, 2, 3...）を提示する
   - 各方針のメリット・デメリットを説明する
   - **ユーザーの判断・承認を待つ**

3. **承認後に修正作業を実施**
   - ユーザーが修正方針を選択・承認してから作業開始
   - 修正内容を明確に説明してから実行
   - 修正中は進捗を報告する

### 2. 原因調査と分析

**原因調査の手順**:
1. **問題の再現**: 問題を再現できる環境を構築
2. **ログの確認**: エラーログやアプリケーションログを確認
3. **コードの確認**: 関連するコードを確認
4. **環境の確認**: 環境設定や依存関係を確認
5. **原因の特定**: 根本原因を特定

**原因分析の報告フォーマット**:
```markdown
## 🔍 原因分析結果
- 問題: [具体的な問題]
- 原因: [特定された原因]
- 影響範囲: [影響を受ける範囲]
```

### 3. 修正方針の提示

**修正方針の提示フォーマット**:
```markdown
## 🔧 修正方針の提案

### Option 1: [方針名]
- 内容: [具体的な修正内容]
- メリット: [利点]
- デメリット: [欠点]

### Option 2: [方針名]
- 内容: [具体的な修正内容]
- メリット: [利点]
- デメリット: [欠点]

どの方針で進めますか？
```

**修正方針の検討ポイント**:
- 修正の影響範囲
- 修正の難易度
- 修正のリスク
- 修正後の保守性

### 4. 禁止事項

**❌ 禁止事項**:
- **いきなり修正作業を開始すること**
- **原因調査なしに推測で修正すること**
- **ユーザーの承認なしにコードを変更すること**
- **複数の修正を同時に実施すること**

## 具体的な指針

### 新機能開発のプロセス

**新機能開発の手順**:
1. **要件定義**: 要件を明確にする
2. **設計**: 設計を検討する
3. **実装**: コードを実装する
4. **テスト**: テストを追加する
5. **レビュー**: コードレビューを実施する
6. **デプロイ**: デプロイを実施する

### バグ修正のプロセス

**バグ修正の手順**:
1. **再現テストの作成**: バグを再現するテストを作成
2. **原因調査**: バグの原因を特定
3. **修正方針の提示**: 修正方針を提示して承認を得る
4. **修正の実装**: 修正を実装
5. **テストの追加**: 修正を検証するテストを追加
6. **レビュー**: コードレビューを実施
7. **デプロイ**: デプロイを実施

### リファクタリングのプロセス

**リファクタリングの手順**:
1. **現状分析**: リファクタリング対象を分析
2. **設計の検討**: 改善後の設計を検討
3. **リファクタリング計画**: 段階的なリファクタリング計画を立てる
4. **テストの確保**: 既存のテストが動作することを確認
5. **段階的な実施**: 小さなステップでリファクタリング
6. **テストの維持**: テストを維持しながらリファクタリング

## 例外・特別なケース

### 緊急の修正

緊急の修正が必要な場合:
- 最小限の変更に留める
- 後で適切な修正を実施
- 原因調査は簡略化しても良い

### レガシーコードへの対応

既存のコードベースに導入する場合:
- 段階的なプロセスの適用
- 既存コードへの影響を最小化
- 移行計画を立てる
