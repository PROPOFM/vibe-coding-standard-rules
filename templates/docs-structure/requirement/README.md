# 仕様書・要件定義（Requirement）

このフォルダは、仕様検討したものを具体的に仕様書・要件定義化する際の情報を保存します。

## 目的

- 機能の要件定義書
- API仕様書
- データモデル定義
- 非機能要件の定義

## ファイル命名規則

- `[feature-name].md`: 機能名で命名
- 例: `user-authentication-requirements.md`, `api-specification.md`

## 推奨フォーマット

```markdown
# [機能名] 要件定義

## 概要
機能の概要と目的

## 機能要件

### 要件1: [要件名]
- **説明**: [内容]
- **優先度**: [高/中/低]
- **受入基準**: [基準]

### 要件2: [要件名]
- **説明**: [内容]
- **優先度**: [高/中/低]
- **受入基準**: [基準]

## 非機能要件

### パフォーマンス
- レスポンス時間: [要件]
- スループット: [要件]
- 同時接続数: [要件]

### セキュリティ
- 認証方式: [要件]
- 認可方式: [要件]
- データ暗号化: [要件]

### 可用性
- 稼働率: [要件]
- 障害復旧時間: [要件]

## 技術要件
- 技術スタック: [内容]
- 依存関係: [内容]
- 外部サービス: [内容]

## データモデル
- エンティティ1: [説明]
- エンティティ2: [説明]

## API仕様（該当する場合）
- エンドポイント1: [説明]
- エンドポイント2: [説明]

## UI/UX要件（該当する場合）
- 画面1: [説明]
- 画面2: [説明]

## 参考資料
- [リンク1](url)
- [リンク2](url)

## 変更履歴
- 2024-01-15: 初版作成
- 2024-01-20: 要件2を追加
```

## 運用方針

### 要件定義の作成タイミング
- `consideration/`での検討が完了した後
- 実装開始前に必ず作成
- 要件変更時は履歴を記録

### レビュー
- ステークホルダーによるレビュー
- 技術的な実現可能性の確認
- 優先度の確認
